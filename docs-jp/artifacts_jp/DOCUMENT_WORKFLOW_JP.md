# ドキュメントワークフロー - 運用ガイドライン

```yaml
document_type: "workflow"
target_audience: "ai_agents"
optimization: "process_consistency"
language: "english"
```

> **注**: このファイルは `artifacts/DOCUMENT_WORKFLOW.md` の日本語訳です。AIエージェントが読むべき正式版は `artifacts/DOCUMENT_WORKFLOW.md`（英語）です。この翻訳は人間による内容確認用です。

本ドキュメントはドキュメント戦略を適用するための運用手順を定義する:
新規プロジェクトのセットアップ、既存プロジェクトへの戦略導入、開発中のドキュメント更新、本戦略の再読み込みタイミング。

```yaml
ownership_split:
  this_doc: "FLOW — いつ行動するか、どの手順に従うか、いつ確認するか"
  FILE_AND_STRUCTURE.md: "HOW + WHERE — ファイル役割、ディレクトリ配置、トークン制限"
  DOCUMENTATION_PHILOSOPHY.md: "WHY — ルールの背後にある原則"
  INDEX.md: "上記すべてへのルーティング"
```

---

## ユースケース

```yaml
use_cases:
  1_new_project: "ゼロから戦略を適用 — 既存のドキュメント構造がない。"
  2_existing_project: "既にドキュメントがあるプロジェクトに戦略を導入（ブラウンフィールド）。"
  3_ongoing_updates: "プロジェクトは既に本戦略に従っている; 開発中にドキュメントを更新または追加。"
```

## ユースケース1: 新規プロジェクトセットアップ

**いつ:** AIエージェントを使用する新規プロジェクトを開始する時。

### ステップ1: エージェントエントリファイル作成

```yaml
action: "プロジェクトが使用する各AIツールにつき1つのエントリファイルを作成する。"
files:
  CLAUDE_md: "Claude Codeを使用する場合"
  AGENTS_md: "DevinまたはCodexを使用する場合"
  GEMINI_md: "Geminiを使用する場合"
token_limit: 200
content_template:
  essential:
    - "role: 本プロジェクトにおけるエージェントの責務"
    - "constraints: エージェントが従うべきプロジェクト固有ルール"
    - "emergency_action: 意図が不明な場合の対応"
  routing:
    - "primary_ref: documents/agents/project.md（または documents/PROJECT.md）"
    - "task_routing: タスクタイプから特定ドキュメントへのマッピング"
  efficiency:
    - "focus_files: エージェントが優先すべきglobパターン"
    - "current_priority: 現在の開発フォーカス"
```

### ステップ2: documentsディレクトリ作成

```yaml
action: "ディレクトリツリーを作成する。"
structure:
  - "documents/（共有仕様）"
  - "documents/agents/（AI向けドキュメント、英語）"
  - "documents/users/（人間向けドキュメント、日本語、_JP.mdサフィックス）"
rule: "内容を投入するディレクトリのみ作成する。投機的に空ディレクトリを作成しない（YAGNI）。"
```

### ステップ3: PROJECT.md作成

```yaml
action: "プロジェクト概要ファイルを作成する。"
placement: "documents/agents/project.md（推奨）または documents/PROJECT.md"
token_limit: 800
content:
  - "プロジェクト名、ステータス、優先度"
  - "技術スタックとアーキテクチャ"
  - "制約（テストカバレッジ、セキュリティ、コンプライアンス）"
  - "現在のフェーズとフォーカス領域"
  - "タスク固有ドキュメントへのルーティング"
```

### ステップ4: README.md作成

```yaml
action: "人間向け入口を作成する。"
language: "プロジェクトの人間向け言語（日本語が主対象）"
content:
  - "プロジェクトの説明と目的"
  - "セットアップと使用方法の手順"
  - "コントリビューションガイドライン"
  - "プロジェクト背景"
token_limit: なし
```

### ステップ5: タスク固有ドキュメントを必要に応じて追加

```yaml
action: "プロジェクトの成長に合わせてドキュメントを作成する — 一度にすべてではない。"
trigger: "PROJECT.md（800トークン）に収まらないコンテキストをタスクが要求する時、タスク固有ファイルに抽出する。"
placement: "documents/agents/<topic>.md（AI向け）または documents/users/<topic>_JP.md（人間向け）"
rule: "重複する内容の多い多数ファイルより、明確なルーティングのある少数ファイルを優先する。"
```

## ユースケース2: 既存プロジェクト導入（ブラウンフィールド）

**いつ:** プロジェクトには既にドキュメントがあるが、本戦略を導入したい場合。

### ステップ1: 既存ドキュメントの監査

```yaml
action: "既存の全ドキュメントを読み、各ファイルを分類する。"
classification:
  ai_facing: "開発中にAIエージェントが必要な内容（仕様、制約、アーキテクチャ）"
  human_facing: "人間の開発者向けの内容（セットアップ、チュートリアル、背景）"
  shared: "両方の読者が必要な内容（APIスキーマ、データモデル）"
  obsolete: "古いまたは冗長な内容"
```

### ステップ2: 新構造へのマッピング

```yaml
action: "各情報がどこに属するかを決定する。"
mapping:
  ai_facing: "documents/agents/（英語、サフィックスなし）"
  human_facing: "documents/users/（日本語、_JP.mdサフィックス）"
  shared: "documents/（コンテキスト依存の言語）"
  obsolete: "削除またはアーカイブ — 移行しない"
```

### ステップ3: エージェントエントリファイル作成

```yaml
action: "CLAUDE.md / AGENTS.md / GEMINI.md を必要に応じて作成する。"
note: "これらは以前存在しなかった — 新規プロジェクトセットアップの手順に従って新規作成する。"
```

### ステップ4: PROJECT.md作成または統合

```yaml
action: "既存ドキュメントからプロジェクトレベルのコンテキストをPROJECT.mdに抽出する。"
token_limit: 800
rule: "既存ドキュメント全体をPROJECT.mdにコピーしない。本質的なコンテキストを抽出し、詳細へルーティングする。"
conflict: "既存ドキュメントが互いに矛盾する場合、ユーザーに矛盾をフラグする — 黙って一方を選ばない。"
```

### ステップ5: ドキュメントの移行

```yaml
action: "既存ドキュメントを新しいディレクトリ構造に移動または書き直す。"
rules:
  - "元が散文中心の場合、トークン効率のために書き直す。"
  - "適切な場合、構造化データをYAMLブロックに変換する。"
  - "重複を排除する: 2つのファイルが同じトピックをカバーしていた場合、1つに統合し他方からリンクする。"
  - "情報を保持する — ユーザー確認なしに内容を削除しない。"
  - "移動・統合・廃止フラグを付けたものを報告する。"
```

### ステップ6: 相互参照の更新

```yaml
action: "移行によって壊れた内部参照を修正する。"
check: "エージェントエントリファイルとPROJECT.mdの全ルーティングパスが正しい場所を指している。"
```

### ブラウンフィールドガード

```yaml
guard:
  scope: "ドキュメント移行が内容の書き直しに拡大しないようにする。"
  rule: "構造的移行と内容改善は別のタスクである。一方を行ってから他方を行う — 同時には行わない。"
  violation_handling: "戦略に違反する既存ドキュメントを見つけた場合、報告に記録する。タスクが明示的に求めない限り、黙って修正しない。"
  local_convention: "明示的なプロジェクト慣習（既存のlint、ハウススタイル）は、競合する場合、本戦略より優先される。矛盾を一度報告し、その後ローカルルールに従う。"
```

## ユースケース3: 継続的ドキュメント更新

**いつ:** プロジェクトは既に本戦略に従っており、開発中にドキュメントの更新が必要な場合。

### いつ更新するか

```yaml
update_triggers:
  architecture_change: "PROJECT.mdと影響を受けるタスク固有ドキュメントを更新する。"
  new_feature: "documents/agents/ にタスク固有ドキュメントを必要に応じて追加; PROJECT.mdのルーティングを更新。"
  constraint_change: "PROJECT.mdの制約セクションと、制約がエージェントの動作に影響する場合はエージェントエントリファイルを更新。"
  tech_stack_change: "PROJECT.mdのtech_stackを更新; 既存ドキュメントがまだ正確か確認。"
  directory_restructure: "エージェントエントリファイルとPROJECT.mdの全ルーティング参照を更新。"
```

### 何を更新するか

```yaml
decision_tree:
  question: "変更はAIエージェントが知る必要がある内容に影響するか？"
  yes:
    action: "関連するAI向けドキュメントを更新する。"
    check: "情報は既にPROJECT.mdにあるか、タスク固有ファイルが必要か？"
    in_project_md: "PROJECT.mdを更新（800トークン以下を維持）。"
    needs_own_file: "documents/agents/<topic>.md を作成または更新し、PROJECT.mdからルーティングを追加。"
  no:
    action: "必要な場合、人間向けドキュメント（documents/users/）またはREADME.mdのみ更新。"
    ai_docs: "AI向けドキュメントは変更しない。"
```

### 更新規律

```yaml
rules:
  - "変更トリガー: スケジュールではなく、コードが変更された時にドキュメントを更新する。"
  - "比例的: 1行のコード修正に完全なドキュメントレビューは不要。"
  - "ルーティング優先: 新規ドキュメントを追加する場合、PROJECT.mdまたはエージェントエントリファイルにルーティング参照を追加する。"
  - "トークンチェック: 更新でファイルがトークン制限を超える場合、分割する — 無制限に成長させない。"
  - "SSOTチェック: 情報を追加する場合、既存ドキュメントと重複しないか確認する。重複する代わりにリンクする。"
```

## ドキュメント作成デシジョンツリー

```yaml
question_1: "この情報は開発中にAIエージェントに必要か？"
  no: "documents/users/（人間向け）またはREADME.mdに配置。"
  yes: "質問2へ続く。"

question_2: "プロジェクトレベルのコンテキストか（概要、技術スタック、制約、ステータス）？"
  yes: "PROJECT.mdに配置（800トークン以下の場合）または要約+リンクを抽出。"
  no: "質問3へ続く。"

question_3: "共有仕様またはスキーマか（API、データモデル）？"
  yes: "documents/（共有ルート）に配置。"
  no: "質問4へ続く。"

question_4: "タスク固有のAI向け詳細か（ワークフロー、最適化ガイド、開発標準）？"
  yes: "documents/agents/<topic>.md に配置。"
  no: "再評価 — 人間向けの可能性がある。"

anti_pattern: "小さな情報ごとに新規ファイルを作成しない。既存ファイルに新しいセクション+ルーティング更新を追加することを優先する。"
```

## 再読み込みトリガー

AIエージェントが行動前に本戦略を再読み込みすべきタイミング。

```yaml
must_re_read:
  - "本戦略を使用するプロジェクトへの初回接触（最初にINDEX.mdを読む）。"
  - "documents/ディレクトリツリーの作成または再構築。"
  - "AIエージェントを使用する新規プロジェクトのセットアップ。"
  - "既存プロジェクトへの本戦略導入（ブラウンフィールド）。"

should_re_read:
  - "新しいエージェントエントリファイルの追加（例: プロジェクトが新しいAIツールを使用開始）。"
  - "ドキュメントの再構築（ディレクトリ間でファイルを移動）。"
  - "プロジェクトを単一から階層に変更（またはその逆）。"
  - "情報がどこに属するか不確実な場合。"

no_re_read_needed:
  - "既存ファイル内の日常的な内容更新。"
  - "確立されたディレクトリへの新しいタスク固有ドキュメントの追加。"
  - "PROJECT.mdの制約やステータスの更新。"
```

## 確認ゲート

ドキュメント構造を変更する前に、影響を評価する。

```yaml
L0_internal: "既存ファイル内の内容更新（構造変更なし） — 進める。"
L1_local: "既存ディレクトリへの新しいタスク固有ファイルの追加 — 進めて報告する。"
L2_structural: "ディレクトリ間でファイルを移動、ルーティングパスの変更、ファイル名変更 — タスクによって明確に暗示される場合のみ進める; 明示的に報告する。"
L3_breaking: "プロジェクトを単一から階層に変更、エージェントエントリファイルの削除、またはdocuments/ツリー全体の再構築 — 実装前に必ず確認する。"
rule: "迷った場合はユーザーに聞く。ドキュメントの構造変更は将来の全AIエージェントセッションに影響する。"
```

## 運用ガイドライン

```yaml
version_control: "ドキュメントはソースコードと共にgitで管理される。"
atomicity: "ドキュメントは原子性を維持する必要がある — 関連するすべてのドキュメントを1コミットで一緒に更新する。ドキュメント更新をブランチ間で分割しない; メインブランチと異なるドキュメント状態を持つブランチは欠陥であり、機能ではない。"
no_branch_for_docs: "ドキュメントのみの変更にブランチを作成しない。作業ブランチに直接コミットする。ブランチ作成はコード開発の関心事（design-principles AI_WORKFLOW.mdを参照）であり、ドキュメントの関心事ではない。"
update_frequency: "変更トリガー — コードやアーキテクチャが変更された時にドキュメントを更新する。"
maintenance: "コードレビュー中にドキュメントをレビューする。PRがアーキテクチャを変更する場合、PROJECT.mdも更新すべき。"
document_sync: "AIエージェントの制約 'update_related_documents_when_changing_project' がこれを強制する。"
reporting: "ドキュメントを更新する際、何が変更されたか、なぜ、ルーティング参照が更新されたかどうかを報告する。"
```
