# ドキュメント哲学 - AIエージェント参考資料

```yaml
document_type: "documentation_philosophy"
target_audience: "ai_agents"
optimization: "token_efficiency"
language: "english"
core_value: "AI_Agent_Priority"
```

> **注**: このファイルは `artifacts/DOCUMENTATION_PHILOSOPHY.md` の日本語訳です。AIエージェントが読むべき正式版は `artifacts/DOCUMENTATION_PHILOSOPHY.md`（英語）です。この翻訳は人間による内容確認用です。

## 核心哲学: AIエージェント優先

ドキュメントは、AIエージェントが最小のトークンコストで正しいコンテキストを見つけられるように構造化される。人間向けドキュメントは別の明確にスコープされた関心事であり、デフォルトではない。

```yaml
principle: "AIエージェント優先"
definition: "ドキュメント構造を人間の可読性ではなく、AIエージェントの処理効率のために最適化する。"
formula: "効率的なコンテキスト = 最小トークン × 正しいルーティング × 低い認知負荷"
rationale: |
  AIエージェントはトークン制限と処理コスト制約の下で動作する。
  散文、冗長性、誤ったルーティング検索に費やされる無駄なトークンは
  パフォーマンスを低下させ、コストを増加させる。
  ドキュメントを構造化し、エージェントが可能な限り少ないトークンで
  正しい情報に到達できるようにする。
```

## 設計優先順位

```yaml
priority_order:
  1: "AIエージェント処理効率（人間の可読性より優先）"
  2: "トークン最適化（タスクあたりの消費を最小化）"
  3: "認知負荷削減（明確なルーティング、曖昧さなし）"
  4: "唯一の情報源（重複排除）"
  5: "汎用性（単一および階層プロジェクト、あらゆる規模）"
```

2つの優先順位が競合する場合、番号が小さい方が優先される。例: 重複排除（4）は良いが、一般的なタスクでAIに3つのファイルを読ませることになるなら、（2と3に違反するため）行わない。

## トークン最適化

```yaml
principle: "タスクあたりのトークン消費を最小化する。"
guidelines:
  - "エントリファイル（agent.md）は200トークン以下 — ルーティングのみで説明しない。"
  - "プロジェクトファイル（PROJECT.md）は800トークン以下 — 1回の処理パスに十分な量。"
  - "構造化情報には散文より密なYAMLを優先する。"
  - "インライン化する代わりに詳細へリンクする。"
  - "1文+リンクが、1段落の再述に勝る。"
  - "装飾的な言語、冗長なヘッダー、前置きをしない。"
```

## 認知負荷削減

```yaml
principle: "AIエージェントが情報の場所を推測することは決してないべき。"
mechanisms:
  routing: "全エントリファイルは次のファイルへの明示的なルーティングを含む。"

  gradual_disclosure: "エントリ → プロジェクト概要 → タスク固有の詳細。すべてを前段に積まない。"
  no_split_by_audience: "全AI向けドキュメントは英語 — 言語はルーティングの軸ではない。"
```

## 唯一の情報源（重複排除）

```yaml
principle: "各情報は正確に1箇所に存在する。"
rules:
  - "概念が2つのファイルに現れる場合、一方が所有し他方はリンクする。"
  - "再述は最大1文 + 所有者へのリンクとしてのみ許可される。"
  - "ドキュメント間に矛盾が生じた場合、所有ドキュメントの記述が権威である。"
  - "ドキュメントが所有していない概念が必要な場合、リンクする — 再導出しない。"
  - "トピックを「なぜ」と「どう」に分割することは許可される — SSOTポインタで結合する。"
```

## フォーマット優先順位

```yaml
principle: "AI効率 > 人間の可読性。"
base_format: "markdown + 埋め込みYAMLブロック"
rationale: |
  YAMLブロックは密でパースしやすく、散文のオーバーヘッドなしに構造を運ぶ。
  Markdownはナビゲーション用の読みやすいセクションヘッダーを提供する。
  JSONは内容がマシンスキーマ（OpenAPI等）の場合にのみ使用する。
  人間が読める散文はREADME.md — 唯一の人間向け例外 — に予約される。
```

## design-principlesとの関係

本アーティファクトセットには兄弟セット `design-principles` が存在する。両者は異なるドメインを持つ独立したアーティファクトセットだが、対象プロジェクトでは通常両方を使用する。

### ドメイン境界

```yaml
this_artifact_set:
  name: "documentation-strategy"
  domain: "ドキュメント（documents/, docs/, README.md, エージェントエントリファイル）"
  owns: "AIエージェント向けドキュメントをどう構造化・ルーティング・保守するか"
  question_answered: "この情報はどこに置き、エージェントはどう見つけるか？"

sibling_artifact_set:
  name: "design-principles"
  domain: "コード（src/, tests/, packages/ 等）"
  owns: "AIエージェントが生成するコードをどう設計・記述するか"
  question_answered: "コードはどのパターン、境界、コントラクトに従うべきか？"
```

### 使用パターン

両セットは対象プロジェクトに配置され、AIエージェントに読まれることを想定している。2つの使用パターンが想定される:

```yaml
pattern_1_independent:
  description: "ユーザーがドメイン固有のタスクで一方のセットを参照する。"
  examples:
    - "コードのバグ修正 → design-principles/ を参照"
    - "プロジェクトドキュメントの更新 → documentation-strategy/ を参照"

pattern_2_combined:
  description: "ユーザーがプロジェクト全体のタスクで両セットを一度に参照する。"
  examples:
    - "'@documents/artifacts/ に従ってこのプロジェクトを開発してください'（両フォルダ）"
    - "プロジェクトのAGENTS.mdが両アーティファクトセットを参照として記載"
  implication: "AIエージェントは両方のコンテキストを同時に保持する。"
```

### 共有概念（意図的な重複）

一部の概念は両セットに現れる。これは意図的である — 各セットが独自のドメインに独立して概念を適用する。両方が同時に読まれる場合、エージェントはタスクに一致するドメインのセットの概念を適用する。

```yaml
shared_concepts:
  single_source_of_truth: "両者は独自のアーティファクト向けにSSOTを定義する。競合しない — それぞれが独自のドキュメントセットを所有する。"
  brownfield_policy: "両者は既存の非準拠作業の取り扱いを定義する。コード違反はdesign-principlesに従う; ドキュメント違反はdocumentation-strategyに従う。"
  confirmation_gate: "両者はL0〜L3の深刻度レベルを定義する。コード変更はdesign-principlesのゲートを使用; ドキュメント変更はdocumentation-strategyのゲートを使用。"
  operational_discipline: "両者は報告・確認ルールを定義する。タスクドメインに一致するものを適用する。"
  proportionality: "両者は影響範囲に応じて労力を調整する。それぞれ独自のドメイン内で。"
```

### 意図的に異なる方針（意図的な相違）

ドキュメントとコードはバージョン管理のニーズが異なるため、一部の方針は意図的に両セット間で異なる。

```yaml
divergent_policies:
  version_control:
    design_principles: "コード変更はブランチを使用する場合がある。デフォルトブランチにいる場合、コミット前にブランチを作成する。"
    documentation_strategy: "ドキュメント変更は作業ブランチに直接コミットする。ドキュメントのみの変更にブランチを作成しない。"
    rationale: "ドキュメントは原子性を要求する — 関連するすべてのドキュメントが1コミットで一貫性を保つ必要がある。メインブランチと異なるドキュメント状態を持つブランチは欠陥であり、機能ではない。コードは対照的に、レビューとロールバックのためにブランチ分離の恩恵を受ける。"
  reporting_language:
    note: "両セットとも報告言語についてはプロジェクトのグローバルエージェントルール（AGENTS.md等）に従う。いずれもグローバルコミュニケーション方針を上書きしない。"
```

### 両方が同時に読まれる場合

```yaml
guidance:
  domain_routing: "タスクがコード（src/, tests/）に触れる場合、design-principlesに従う。タスクがドキュメント（documents/, README.md）に触れる場合、documentation-strategyに従う。"
  mixed_tasks: "タスクがコードとドキュメントの両方に触れる場合（例: 機能追加 AND PROJECT.md更新）、各セットをそれぞれのドメインに適用する。ドメイン間でルールを混ぜない。"
  conflict_resolution: "概念が両セットに存在する場合、現在のアクションに一致するドメインのセットが権威である。共有概念（SSOT、Brownfield、Gate）はドメインごとに適用され、統合されない。"
```

## 誤読防止

これらの原則は過剰に解釈されやすい。以下のように短縮しないこと:

```yaml
misreadings:
  - "'AI効率 > 人間の可読性' != 人間は重要でない（READMEは明示的に人間向け; ルールはAI向けファイルにのみ適用）"
  - "'トークン最適化' != すべてを1ファイルにする（ルーティングと認知負荷も重要 — 設計優先順位を参照）"
  - "'唯一の情報源' != 概念を2度言及してはならない（1文の再述 + リンクは許可される）"
  - "'エントリファイル ≤200トークン' != エージェントに情報を飢えさせる（エントリファイルは詳細へルーティングする; エントリファイル自体が詳細ではない）"
  - "'汎用性' != 1つの固定テンプレート（戦略は単一プロジェクトと階層プロジェクトに適応する; 構造はプロジェクト形状に従う）"
  - "'YAMLブロック推奨' != すべての散文を禁止する（説明と例はmarkdownを使用; YAMLは構造化データ用）"
  - "'documents/agents/ はAI専用' != 人間は読めない（読める; AI向けに最適化されているだけで人間が読むことを制限しない）"
  - "'子の独立性' != 子は親と協調できない（必要な場合、親を外部プロジェクト参照として扱う）"
  - "本戦略 != ドキュメント生成器（構造とルーティングを定義する; AIエージェントがこれらのルールに従って内容を記述する）"
```
